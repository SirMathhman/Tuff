# Language Spec — One Page ⚙️🔒

## 1) Purpose & Goals

- **Name (working):** Untitled systems language
- **Primary goal:** Prevent unsafe code generated by agents for systems, embedded, and multi-platform targets (avoid buffer overflows, use-after-free, data races, and memory leaks) while enabling reliable cross-target development workflows.
- **Priorities:** **Safety** (primary), **Performance**, **Expressiveness**, **Interop (C/LLVM)**.
- **Guiding priority:** **Safety** — trade-offs favor strong static guarantees even when they affect ergonomics or code size.

---

## 2) Design Summary

- **Typing:** Static, **dependent/linear** types with ML-like inference.
- **Memory:** **Zero-runtime** with **linear/move** semantics (no GC). Ownership is enforced by the type system; runtime checks are only added when explicitly requested (e.g., `--sanitize`). Pointers and slices: slices use `*[T]`; mutable raw pointers use `*mut T`. Raw pointers are **guaranteed aligned and non-null** by construction.
- **Verification:** **SMT-backed**, mostly-inferred proofs; `require`/`ensure`/`assert` annotations for explicit invariants.
- **Concurrency:** Ownership-based across threads + linear types to statically prevent data races. The **async/futures** model is the primary concurrency abstraction to be supported and verified in the stdlib; message-passing channels and actor frameworks can be layered on top.
- **Interop/Targets:** `extern` C-style FFI with **annotated verification contracts** (pre/post conditions checked at compile time). Primary backends: LLVM and JS. Supported targets include bare-metal, Linux, Windows, WebAssembly, Node.js, and RTOS. The compiler includes first-class cross-compilation tooling, per-target stdlib slices, and platform-specific ABI handling to support seamless multi-platform development and CI workflows.
- **Ergonomics:** Expression-oriented, **braced** blocks, colon-style type annotations, concise syntax, and a focused standard library. Naming conventions: identifiers use `camelCase`; type, module, and constructor names use `PascalCase`. Visibility: items are **private by default**; use `out` to export individual items (e.g., `out fn foo()`). Method receivers (`this`) are inferred and optional; use `*this` for an explicit by-value receiver or `&this` for an explicit borrow when desired. Return types are inferred and may be omitted; the compiler infers the result type and any verification obligations are still checked.

> Note: The compiler prioritizes helpful diagnostics and fast incremental cycles; proofs are inferred automatically and unresolved obligations produce clear diagnostics.

## 3) Core Syntax (examples)

- Bindings & functions:
  - `let x : I32 = 0;`
  - `fn add(a : I32, b : I32) => a + b; // return type inferred`
- Structs / methods:
  - `struct Point { x : I32; y : I32; }`
  - `module Point { fn manhattan(*this) : I32 => abs(this.x) + abs(this.y); }`
- Externs:
  - `extern fn c_get() : I32;`
- Match / enums:
  - `enum Opt<T> { None, Some(T) }`
  - `match opt { Some(v) => v, None => 0 }`
- Verification annotations:
  - `fn safeRead(buf: &Buffer, idx: USize) : I8 requires idx < buf.len ensures true => ...`
- Pointer/slice and inline refinement example: `fn readByte(bytes : *[I8], index : USize < bytes.init) : I8 => bytes[index];`

---

## 4) Type System & Safety

- **Linear types / move semantics** for resources: values with linear kind must be consumed/moved; compiler enforces single-owner rules.
- **Dependent fragments**: types can depend on values for sizes and bounds, e.g., `Buffer[n : USize]`.
- **SMT integration:** The solver discharges assertions and refinement obligations; the compiler **infers proofs automatically**. Return types may be inferred by the compiler when omitted; inferred return types and their verification obligations are treated the same as explicit types. If a required proof cannot be inferred, compilation fails with clear diagnostics and suggested next steps.
- **Safety properties enforced statically:** bounds, lifetime/moves (no use-after-free), race-freedom (no shared mutable aliasing), memory-leak freedom by default patterns + optional proof obligations for complex lifetimes.

---

## 5) Memory & Concurrency Model

- **No GC**, minimal runtime support for platform-required demarcation only.
- **Ownership model:** linear/move with **auto-borrow and inference by default**; explicit move/borrow annotations are available for precise control. Inference is predictable and silent by default; diagnostic reporting can be enabled via project settings or compiler flags.
- **Message-passing / channels** in stdlib built on top of ownership rules (safe by construction).

---

## 6) Verification Workflow

- **Full verification by default**: proofs for dependent and linear obligations are required and failures block compilation.
- **Proof model**: proofs are inferred automatically; source-level proof hints are omitted by design. The compiler provides clear diagnostics and suggested remedies when obligations cannot be discharged.
- **Selective verification**: bypassing full verification is restricted and requires explicit compiler flags; this is discouraged for production code.
- **Runtime checks**: optional `--sanitize` mode inserts runtime checks when proofs are incomplete or when building with a non-default verification mode.

---

## 7) Tooling & Ergonomics

- Fast incremental compiler, **LSP** server, formatter, and REPL.
- Multi-platform development: cross-compilation toolchains, per-target build profiles, cross-target verification, and CI integration.
- Errors: concise primary message + short actionable fix suggestions + `--explain` for extended diagnostics.
- Small but useful standard library with **buffers & slices prioritized for a verified bootstrap** (minimal API first), followed by a **full verified allocator** and device and concurrency primitives (async/futures).

## 8) Minimal EBNF (core)

```
program     ::= { item }
item        ::= fn_decl | struct_decl | enum_decl | extern_decl
fn_decl     ::= "fn" ident "(" [param_list] ")" [":" type] "=>" expr ";"
param_list  ::= param { "," param }
param       ::= ident ":" type [ "<" refinement ">" ]
refinement  ::= expr  // solver-backed boolean expression over parameters and expressions
type        ::= "*" "[" type "]"        // slice type: *[T]
           |  "*" type                       // pointer to single value: *T
           |  ident                            // nominal types; generics handled elsewhere
struct_decl ::= "struct" ident "{" { field } "}"
field       ::= ident ":" type ";"
enum_decl   ::= "enum" ident "{" { variant } "}"
extern_decl ::= "extern" "fn" ident "(" [param_list] ")" ":" type ";"
expr        ::= literal | ident | "match" expr "{" case_list "}" | block | call
```

**Naming conventions:** identifiers are `camelCase`; type, module, and constructor names are `PascalCase`.

**Generics:** use angle-bracket syntax, e.g., `Opt<T>` and `fn f<T>(x: T)`; generic constraints use inline `:` notation, e.g., `fn f<T: Trait>(x: T)`.

## Bootstrap compiler (limited initial scope)

- The bootstrap compiler implements a small, well-defined subset of the language to enable early verification and iteration.
- Initial feature set includes: core expression syntax, function/struct/enum declarations, a minimal dependent fragment (fixed-size buffers/slices), linear move semantics with auto-borrow inference, SMT-backed verification limited to bounds and ownership obligations, and a minimal verified stdlib (buffers & slices, basic indexing and length proofs).
- A compact, simple allocator (for example, a verified bump allocator) is included in the bootstrap scope where feasible; the full allocator will be introduced as verification and implementation capacity grows.
- Backends: initial bootstrap targets are LLVM native and JS/Node; cross-compilation support and additional targets will be added incrementally.
- Policy: the bootstrap compiler enforces verification for implemented features; unsupported language features will be clearly documented and staged for future releases.
- Goal: keep the bootstrap implementation small, well-tested, and verifiable before expanding the language surface.

## Decisions (recorded defaults)

- **Guiding priority:** **Safety** — trade-offs favor strong static guarantees even when they affect ergonomics or code size.
- **Verification default:** **Full** — proofs required by default; bypassing full verification is restricted and requires explicit compiler flags; this is discouraged for production code.
- **Proof model:** compiler-inferred proofs; the compiler omits source-level proof hints by design. If a proof cannot be inferred, compilation fails with clear diagnostics and suggested next steps.
- **Diagnostics:** compiler inference is silent/predictable by default; diagnostic reporting can be enabled via project or compiler settings.
- **Memory ergonomics:** **Auto-borrow/inference** by default; explicit annotations are available for precise control.
- **Concurrency primitive:** **Async/futures** model prioritized for verified concurrency; channels & actor frameworks will be built on top.
- **Allocator:** **Full verified allocator** planned, with a compact verified allocator (e.g., bump allocator) targeted for the bootstrap compiler when feasible.
- **Bootstrap compiler:** **Limited initial feature set** — core syntax, minimal dependent fragments, buffers & slices, simple allocator, and verification limited to supported features.
- **Stdlib bootstrap priority:** **Buffers & Slices** (verified first) — initial API scope: minimal verified slice view + indexing + length/bounds proofs.
- **Toolchain/targets:** **LLVM/clang compatibility prioritized**; initial bootstrap targets: LLVM native and JS/Node. The toolchain will explicitly support **multi-platform development** via cross-compilation, per-target stdlib slices, and CI-ready verification pipelines.
- **Visibility & exports:** items are **private by default**; use `out` to export individual items from a `module`.
- **Generics & constraints:** generics use `<T>` and generic constraints use inline `:` syntax, e.g., `fn f<T: Trait>(x: T)`.
- **Error model:** prefer `Result<T, E>` with module-local `E` types; there is no single required `Error` trait.
- **Pointers & safety:** `*mut T` is the mutable raw pointer form; raw pointers are guaranteed aligned and non-null. **Dereference policy:** whether deref requires `unsafe` or may be proven safe by verification is pending decision.
