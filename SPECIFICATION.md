# Language Spec — One Page ⚙️🔒

## 1) Purpose & Goals

- **Name (working):** Untitled systems language
- **Primary goal:** Prevent unsafe code generated by agents for systems, embedded, and multi-platform targets (avoid buffer overflows, use-after-free, data races, and memory leaks) while enabling reliable cross-target development workflows.
- **Priorities:** **Safety** (primary), **Performance**, **Expressiveness**, **Interop (C/LLVM)**.
- **Guiding priority:** **Safety** — trade-offs favor strong static guarantees even when they affect ergonomics or code size.

---

## 2) Design Summary

> **Note:** This section describes the full Tuff language design (targeting Phase 3+). See **Development Phases** below for details on what's implemented in Phase 1 (Bootstrap) and Phase 2 (Self-hosted Compiler).

- **Typing:** Static, **dependent/linear** types with ML-like inference.
- **Memory:** **Zero-runtime** with **linear/move** semantics (no GC). Ownership is enforced by the type system; runtime checks are only added when explicitly requested (e.g., `--sanitize`). Pointers and slices: slices use `*[T]`; mutable raw pointers use `*mut T`. Raw pointers are **guaranteed aligned and non-null** by construction.
- **Verification:** **SMT-backed**, mostly-inferred proofs; `require`/`ensure`/`assert` annotations for explicit invariants.
- **Concurrency:** Ownership-based across threads + linear types to statically prevent data races. The **async/futures** model is the primary concurrency abstraction to be supported and verified in the stdlib; message-passing channels and actor frameworks can be layered on top.
- **Interop/Targets:** `extern` C-style FFI with **annotated verification contracts** (pre/post conditions checked at compile time). Primary backends: LLVM and JS. Supported targets include bare-metal, Linux, Windows, WebAssembly, Node.js, and RTOS. The compiler includes first-class cross-compilation tooling, per-target stdlib slices, and platform-specific ABI handling to support seamless multi-platform development and CI workflows.
- **Ergonomics:** Expression-oriented, **braced** blocks, colon-style type annotations, concise syntax, and a focused standard library. Naming conventions: identifiers use `camelCase`; type, module, and constructor names use `PascalCase`. Visibility: items are **private by default**; use `out` to export individual items from a module to its parent (e.g., `out fn foo()`), or `out module M { ... }` to export a nested module. **Module structure:** A file can contain multiple modules, and modules can be nested within other modules; file structure is independent of module hierarchy. Method receivers (`this`) are inferred and optional; use `*this` for an explicit by-value receiver or `&this` for an explicit borrow when desired. Return types are inferred and may be omitted; the compiler infers the result type and any verification obligations are still checked. **Method call syntax:** both `obj.method()` and `method(obj)` styles are valid.

> Note: The compiler prioritizes helpful diagnostics and fast incremental cycles; proofs are inferred automatically and unresolved obligations produce clear diagnostics.

## 3) Core Syntax (examples)

- Bindings & functions:
  - `let x : I32 = 0;`
  - `fn add(a : I32, b : I32) => a + b; // return type inferred`
- Structs / methods:
  - `struct Point { x : I32; y : I32; }`
  - `module Point { fn manhattan(*this) : I32 => abs(this.x) + abs(this.y); }`
  - Method calls work both ways: `obj.method()` and `method(obj)` are both valid.
- Externs:
  - `extern fn c_get() : I32;`
- Match / enums (structural only, no guards):
  - `enum Opt<T> { None, Some(T) }`
  - `match opt { Some(v) => v, None => 0 }`
- Error handling:
  - `fn divide(a : I32, b : I32) : Result<I32, E> => ...`
  - Short-circuit with `?`: `let result = divide(10, 2)?;` (propagates on error)
- Verification annotations:
  - `fn safeRead(buf: &Buffer, idx: USize) : I8 requires idx < buf.len() ensures true => ...`
- Pointer/slice and inline refinement example:
  - `fn readByte(bytes : *[I8], index : USize < bytes.len()) : I8 => bytes[index];`
  - Refinements provide context: `index : USize < this.len()` refers to `this.len()`, not a free variable.

## 4) Type System & Safety

- **Linear types / move semantics** for resources: values with linear kind must be consumed/moved; compiler enforces single-owner rules.
- **Dependent fragments**: types can depend on values for sizes and bounds, e.g., `Buffer[n : USize]`.
- **SMT integration:** The solver discharges assertions and refinement obligations; the compiler **infers proofs automatically**. Return types may be inferred by the compiler when omitted; inferred return types and their verification obligations are treated the same as explicit types. If a required proof cannot be inferred, compilation fails with clear diagnostics and suggested next steps.
- **Safety properties enforced statically:** bounds, lifetime/moves (no use-after-free), race-freedom (no shared mutable aliasing), memory-leak freedom by default patterns + optional proof obligations for complex lifetimes.

---

## 5) Memory & Concurrency Model

- **No GC**, minimal runtime support for platform-required demarcation only.
- **Ownership model:** linear/move with **auto-borrow and inference by default**; explicit move/borrow annotations are available for precise control. Inference is predictable and silent by default; diagnostic reporting can be enabled via project settings or compiler flags.
- **Borrow semantics:** Rust-like (one mutable reference XOR multiple immutable references to the same data).
- **Message-passing / channels** in stdlib built on top of ownership rules (safe by construction).
- **Concurrency:** async/futures model planned for full language; bootstrap and phase 2 do not include concurrency primitives.

---

## 6) Verification Workflow

- **Full verification by default**: proofs for dependent and linear obligations are required and failures block compilation.
- **Mandatory verification (Coq-style):** The language enforces proof obligations like Coq, ensuring proofs are provided for all critical invariants. This is fundamental to the language design; verification cannot be disabled in production code.
- **Proof model**: proofs are inferred automatically when possible; source-level proof hints may be required for complex obligations. The compiler provides clear diagnostics and suggested remedies when obligations cannot be discharged.
- **Selective verification (restricted)**: bypassing full verification is restricted to bootstrap development only and requires explicit compiler flags; this is discouraged for production code.
- **Compilation failure on unresolved proofs**: if an obligation cannot be discharged, the compiler fails with a clear diagnostic showing the proof obligation and potential next steps.

## 7) Tooling & Ergonomics

- Fast incremental compiler, **LSP** server, formatter, and REPL.
- Multi-platform development: cross-compilation toolchains, per-target build profiles, cross-target verification (Phase 3+), and CI integration.
- Errors: concise primary message + short actionable fix suggestions + `--explain` for extended diagnostics.
- Standard library strategy: Phase 1 minimal (just syntax), Phase 2 adds simple allocators and Result type, Phase 3+ adds channels, async/futures, buffers/slices, and device primitives.

## 8) Core EBNF (Phase 2+)

```
program     ::= { item }
item        ::= fn_decl | struct_decl | enum_decl | module_decl | extern_decl
fn_decl     ::= ["out"] "fn" ident "(" [param_list] ")" [":" type] "=>" expr ";"
param_list  ::= param { "," param }
param       ::= ident ":" type [ "<" refinement ">"]
refinement  ::= expr  // solver-backed boolean expression over parameters
type        ::= "*" "mut" type                   // mutable pointer: *mut T
           |  "*" "[" type "]"                  // slice type: *[T]
           |  ident                             // nominal types; generics elsewhere
struct_decl ::= "struct" ident "{" { field } "}"
field       ::= ident ":" type ";"
enum_decl   ::= "enum" ident "{" { variant } "}"
variant     ::= ident ["(" type ")"]
module_decl ::= ["out"] "module" ident "{" { item } "}"
extern_decl ::= "extern" "fn" ident "(" [param_list] ")" ":" type ";"
expr        ::= literal | ident | "match" expr "{" { case } "}"
           |  block | call | method_call | binop
method_call ::= ident "." ident "(" [expr_list] ")"  // obj.method()
           |  ident "(" ident "," [expr_list] ")"    // method(obj, ...)
binop       ::= expr ( "+" | "-" | "*" | "/" | "==" | "!=" | "<" | ">" | "&&" | "||" ) expr
```

**Phase 1 (Bootstrap) EBNF differs:**

- No pointers, no slices, no type annotations in some contexts.
- Simpler expressions (arithmetic, comparison, logical operators only).
- No extern declarations.

## 10) Development Phases

### **Phase 1: Bootstrap Compiler (Minimal MVP, written in JavaScript)**

- **Goal:** Rapid prototyping of syntax and core semantics to validate language design choices.
- **Scope:** Core expression syntax, function/struct/enum declarations, basic pattern matching (no guards), assignment as statement only, method calls (both syntaxes).
- **Targets:** JavaScript only (outputs Node.js-compatible code).
- **Type checking:** Minimal (JS-like) to get something working quickly; constraints will be applied in Phase 2.
- **Verification:** None. No SMT proofs, no refinement checks, no dependent types.
- **Concurrency:** Not included.
- **Pointers/FFI:** Not included.
- **Status:** Throwaway implementation; will be discarded after Phase 2 is stable.
- **Output:** Executable Tuff programs in Node.js.

### **Phase 2: Self-Hosted Compiler (written in Tuff, compiles to JS)**

- **Goal:** Establish a stable, production-ready compiler that can compile itself.
- **Scope:** All Phase 1 features + modules (file-independent structure), visibility/exports (`out`), proper ML-style type inference, allocators (simple implementations to start), full error handling (Result + `?` operator).
- **Targets:** JavaScript (Node.js).
- **Verification:** Begin integrating SMT-backed verification for safe operations; full mandatory verification on allocators and resource handling.
- **Concurrency:** Not included initially; async/futures planned for later phases.
- **Pointers/FFI:** Not included in this phase.
- **Output:** Tuff compiler in Tuff; produces JS output.

### **Phase 3+: Full Language (LLVM + Platforms)**

- **Goal:** Support systems programming with full safety guarantees.
- **Scope:** Pointers (raw `*T`, slices `*[T]`), FFI with annotated contracts, dependent types, comprehensive allocators, async/futures, cross-platform support (Linux, Windows, bare-metal, RTOS, WebAssembly), LLVM backend, full verification with proof obligations.
- **Targets:** LLVM (native) and JavaScript; cross-compilation toolchains per target.
- **Features:** Full dependent/linear type system, SMT-backed proofs, ownership-based concurrency, message-passing channels, actor frameworks.
- **Output:** Native code with bulletproof safety or JS as appropriate.

## 11) Decisions (recorded defaults)

- **Guiding priority:** **Safety** — trade-offs favor strong static guarantees even when they affect ergonomics or code size.
- **Verification default:** **Full and Mandatory** — proofs required by default; verification cannot be optionally disabled in production code (Coq-style). Verification is integral to the language, not optional.
- **Proof model:** compiler-inferred proofs; explicit proof hints may be required for complex obligations. If a proof cannot be inferred, compilation fails with clear diagnostics and suggested next steps.
- **Diagnostics:** compiler inference is silent/predictable by default; diagnostic reporting can be enabled via project or compiler settings.
- **Memory ergonomics:** **Auto-borrow/inference** by default (Rust-like borrowing); explicit annotations available for precise control.
- **Dereference safety:** Raw pointers may be dereferenced in safe code when the verifier proves safety; no `unsafe` annotation required for verified-safe dereferences. Unverified dereferences will produce compilation errors with clear diagnostics.
- **Borrow semantics:** Rust-like rule: one mutable reference XOR multiple immutable references to the same data.
- **Module system:** Multiple modules per file, nested modules allowed. File structure independent of module hierarchy. Visibility: private by default; `out` exports to parent module, `out module` exports nested modules.
- **Method syntax:** Both `obj.method()` and `method(obj)` styles are valid and interchangeable.
- **Match expressions:** Structural matching only (no guards) to simplify verification and reduce cyclomatic complexity. Pattern matching is deterministic and exhaustive.
- **Assignment:** Assignment is a statement only, not an expression. `let x = (y = 3);` is invalid.
- **Error handling:** `Result<T, E>` with `?` operator for short-circuiting on error. Errors are composable via `?`.
- **Generics & constraints:** Generics use `<T>` syntax; generic constraints are proved by the verifier rather than using explicit trait bounds. Trait bounds are not a separate system; they are verification obligations.
- **Concurrency primitive:** **Async/futures** model prioritized for Phase 3+ verified concurrency; channels & actor frameworks will be built on top. Not included in Phase 1 or Phase 2.
- **Allocators:** Simple allocators (e.g., arena/bump allocators) targeted for Phase 2 as proof-of-concept; full verified allocator planned for Phase 3+.
- **Phase roadmap:** Phase 1 (Bootstrap in JS, minimal), Phase 2 (Self-hosted Tuff in JS, full verification), Phase 3+ (LLVM, pointers, FFI, full language).
- **Visibility & exports:** items are **private by default**; use `out` to export individual items from a `module`, or `out module` to export a nested module.
- **Pointers & safety:** `*mut T` is the mutable raw pointer form; `*[T]` is the slice/array-pointer form. Raw pointers are guaranteed aligned and non-null by construction. Available in Phase 3+.
- **Stdlibs priority:** Phase 2 focuses on simple allocators; Phase 3+ adds channels, async/futures, and platform-specific libraries.
- **Proof hint syntax:** To be decided in Phase 2; compiler tries to infer proofs automatically first.

## 12) Clarifications & Open Items (from review session)

**Clarified in specification review:**

- Dereference safety: verified-safe dereferences allowed in safe code; no `unsafe` needed.
- Module system: multiple modules per file, nested modules, file-independent hierarchy.
- Borrow semantics: Rust-like (one mutable XOR many immutable).
- Generics: verifier-backed, no separate trait system; constraints are proof obligations.
- Error handling: `match` + `?` operator.
- Refinements: context-aware (e.g., `i: USize < this.len()`).
- Phase roadmap: Phase 1 (JS bootstrap), Phase 2 (self-hosted Tuff), Phase 3+ (LLVM + full language).
- Verification: Mandatory (Coq-style), not optional in production.

**Open items for future specification:**

- Proof hint syntax conventions (Phase 2).
- Detailed FFI contract syntax and cross-module verification rules.
- Async/futures runtime design and ownership-aware executor semantics (Phase 3+).
- Full allocator verification invariants and implementation strategy (Phase 3+).
- Standard library API design for allocators, channels, and platform primitives.
- Cross-compilation and multi-target build infrastructure.
- LLVM code generation strategy and ABI handling per platform.
